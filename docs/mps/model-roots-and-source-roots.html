MPS model roots and source roots

How a module descriptor tells MPS where to find its models: model roots, their content directory, and their source roots.

Source references point at JetBrains MPS commit c4689bd (post-2025.3 master). Line numbers are pinned to that commit.

The three terms

Module
A solution, language, devkit, or generator. Its descriptor file (*.msd, *.mpl, *.devkit, *.mpst) lists the module's models inside a <models> element.
Model root
An entry under <models> that describes one group of related models and where they come from. A model root has a type (for example default), an optional content directory, and a list of source roots.
Content directory
A single base path, written as the contentPath attribute. Source roots that are given as relative paths are resolved against it.
Source root
A directory that MPS reads models from. MPS walks each source root recursively and loads every model file it finds; one source root can hold many models.

In one line: a module's models live in directories called source roots; a model root groups source roots under a type and an optional base directory (contentPath).

Interfaces: ModelRoot.java, SourceRoot.java

How a source root gives its path

A source root's path is written in one of two ways:

  • Relative — a location attribute, resolved against the model root's contentPath. The value location="." refers to the content directory itself.
  • Absolute — a path attribute, taken as written (after macros such as ${module} and ${mps_home} are expanded).

Each source root also has a kind. The SOURCES kind supplies models. The EXCLUDED kind marks a subtree to leave out of scanning.

When MPS loads a default model root, it takes the source roots of the SOURCES kind and reads models from each one. For example, DefaultModelRoot.loadModels() iterates getSourceRoots(SOURCES) and collects the models under each.

kinds: SourceRootKinds.java

How it looks in a module descriptor

A model root persists inside <models>. The XML follows a fixed shape (FileBasedModelRoot.save()):

  • <modelRoot> carries the type and, when a content directory is set, the contentPath.
  • Each SOURCES source root is a <sourceRoot> child; each EXCLUDED one is an <excluded> child.
  • A source root child carries location (relative) or path (absolute).

Relative source root

<modelRoot contentPath="${module}" type="default">
  <sourceRoot location="models" />
</modelRoot>

Models are read from ${module}/models. This is the common form: across MPS-extensions, mbeddr.core and iets3.opensource, location= appears about 1131 times to path='s 29.

Absolute source root

<modelRoot type="java_classes">
  <sourceRoot path="${mps_home}/lib/mps-behavior-runtime.jar" />
</modelRoot>

The source root gives a full macro path and the model root needs no contentPath.

Content directory shared by several source roots

<modelRoot contentPath="${mps_home}/lib" type="java_classes">
  <sourceRoot location="util-8.jar" />
  <PackageScope skip-private="false">
    <include prefix="org.jetbrains.org.objectweb.asm." />
  </PackageScope>
</modelRoot>

Each location resolves against the shared contentPath (${mps_home}/lib/util-8.jar). Some model root types add child elements of their own, such as PackageScope.

examples: behavior-runtime.msd, java.stub.msd

Model root types

Each model root has a type string. These are the types in the MPS tree:

type Class Models come from
default DefaultModelRoot model files you author, stored under the source roots (.mps / .mpsr / .model)
java_classes JavaClassStubsModelRoot compiled Java in the .jar / .class files under the source roots
jdk JDKStubsModelRoot the configured project JDK uses no source roots
java_source_stubs JavaSourceStubModelRoot Java source files under the source roots
kotlin_jvm, kotlin_common Kotlin plugin roots Kotlin artifacts under the source roots
others PSI Java stub (IdeaPlugin), property_files_stubs (sample), … plugin- or sample-specific sources

A default model root holds the models you write and MPS saves to disk. The other types build their models in memory from compiled code or source files each time the module loads; those generated models appear in other models as <import> references. The jdk type takes its input from the project JDK setting, so it configures no source roots.

type constants: PersistenceRegistry.java (DEFAULT_MODEL_ROOT, JAVA_CLASSES_ROOT, JDK_CLASSES_ROOT)

What the real projects contain (git-tracked descriptors): MPS itself — default ~1684, java_classes ~100, and a handful of kotlin_jvm / jdk / kotlin_common / java_source_stubs; MPS-extensions, mbeddr.core and iets3.opensource — default and java_classes.

What this means for the hooks

  • These hooks act on tracked model files, which live in default model roots, so they read type="default" model roots and collect the directories of their SOURCES source roots.
  • To get those directories, resolve each <sourceRoot location=…> against contentPath and take each <sourceRoot path=…> as written, expanding ${module} to the descriptor's directory. A ${mps_home} path points at an MPS installation outside the repository, so the hooks leave it as written and it matches no tracked model.
  • A model belongs to a module when it sits under one of that module's source roots. A model under no source root is what mps-check-orphan-models reports.
  • mps-check-model-naming measures a model's file name against its qualified name relative to the source root the model sits under.
